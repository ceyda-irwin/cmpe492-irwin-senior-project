# CMPE492 – Senior Project

## Project description

This repository implements a **Gray–Scott reaction–diffusion** simulation on a 2D grid and studies **inverse parameter recovery**: given a target spatial pattern (the activator field \(V\)), find or predict the kinetic parameters **feed \(F\)** and **kill \(k\)** that produce a similar pattern.

Two complementary approaches are included:

1. **Evolutionary search** — A genetic algorithm evolves \((F, k)\) candidates against a scalar **fitness** that compares simulated \(V\) to a fixed target. Fitness was refined across several versions (pixel MSE, statistics, FFT spectrum, radial profile, morphological / connected-component features).
2. **Supervised learning** — Synthetic datasets of labeled patterns are generated by sampling \((F, k)\), simulating to a steady-like state, and training a **CNN** to regress \((F, k)\) from the \(V\) image (with evaluation and reconstruction visualizations).

The work is developed for the **CMPE492** senior design course; methodology, figures, and code paths are organized for reproducibility and reporting.

## Objectives

- Simulate the Gray–Scott model with stable numerics (finite differences, clipping, optional early stop on convergence).
- Define reproducible **target** patterns and quantitative **fitness** functions aligned with visual similarity.
- **Optimize** \((F, k)\) with a GA within biologically plausible \(F,k\) ranges and document trade-offs between fitness formulations.
- **Explore** the \((F, k)\) landscape via parameter sweeps (coarse and refined grids) and summarize results (CSVs, heatmaps, contact sheets).
- **Learn** an image-to-parameter mapping with PyTorch and assess generalization on held-out data.

## Methodology (high level)

1. **Target** — `rd/generate_target.py` writes the reference pattern to `outputs/target/` (`target_pattern.npy` + preview PNG).
2. **Sweeps** — `rd/parameter_sweep.py` and `rd/refined_sweep.py` write grids of patterns and metrics under `outputs/sweeps/`; `rd/plot_heatmaps.py` and `rd/make_contact_sheet.py` aggregate results.
3. **GA** — `rd/ga_optimize_fk.py` (baseline fitness), `rd/ga_optimize_fk_v2_radial.py`, `rd/ga_optimize_fk_v3_components.py` (richer fitness) save best individuals and fitness curves under `outputs/ga/`.
4. **Fitness diagnostics** — Scripts named `rd/fitness_param_test*.py` and `rd/*_visual_check.py` probe behavior of specific terms; comparison figures go to `outputs/visual_checks/`.
5. **ML pipeline** — `scripts/generate_ml_dataset.py` builds `data/processed/rd_dataset.npz`; `scripts/train_cnn_regressor.py` / `train_cnn_regressor_v2.py` train models into `scripts/models/`; `scripts/evaluate_cnn_regressor.py` writes reconstructions to `scripts/models/eval_outputs/`.

## Project structure

- **`rd/`** — Reaction–diffusion experiments and reporting helpers:
  - Simulation and targets: `generate_target.py`, `reaction_diffusion.py`
  - Sweeps: `parameter_sweep.py`, `refined_sweep.py`, `plot_heatmaps.py`, `make_contact_sheet.py`
  - GA: `ga_optimize_fk.py`, `ga_optimize_fk_v2_radial.py`, `ga_optimize_fk_v3_components.py`
  - Fitness experiments: `fitness_test.py`, `fitness_param_test.py`, `fitness_param_test_v2.py` … `v5_components.py`
  - Visual checks: `fft_visual_check.py`, `component_visual_check.py`, `radial_profile_check.py`, `compare_target_and_best.py`
- **`outputs/`** — Generated assets (tracked where useful for the report):
  - `outputs/target/` — reference `target_pattern.npy` / `target_pattern.png`
  - `outputs/ga/` — GA snapshots (`best_gen/`, `best_radial_gen/`, `best_components_gen/`) and `ga_fitness_progress*.png`
  - `outputs/sweeps/` — `sweep_outputs/`, `refined_sweep_outputs/` (patterns, `results.csv`, heatmaps)
  - `outputs/visual_checks/` — diagnostic and comparison figures
- **`scripts/`** — ML: dataset generation, training, inspection, evaluation; trained weights and curves under **`scripts/models/`**
- **`data/processed/`** — `rd_dataset.npz` (default output of dataset generation; **gitignored**)

**Important:** Run Python entry points from the **repository root** so `Path(__file__).resolve().parents[1]` resolves to this project and all relative roots (`outputs/`, `data/`, `scripts/models/`) stay consistent.
